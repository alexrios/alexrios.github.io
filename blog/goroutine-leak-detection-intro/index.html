Part of series: Goroutine Leak Detection

Hunting ghost goroutines

17 December 2025 / 8 min read
  • go
  • runtime
  • profiling
  • debugging

New to goroutine leaks? Start with Part 0: When goroutines become orphans for a gentler introduction.

They’re there. Waiting. Forever.

Somewhere in your program, there are goroutines that will never wake up. They’re blocked on channel sends that will never complete, waiting on mutexes that nobody will unlock, stuck in select statements that will never have a case ready. You got the idea.

They’re not dead, the runtime still knows they exist, but they’re not alive either.

They’re ghost goroutines, consuming memory, holding onto resources, and slowly accumulating until your program runs out of steam.

And here’s the problem: your regular goroutine profiler can’t tell you which ones they are.

Go 1.26 introduces an experimental goroutine leak detector that changes this.

The problem: waiting for a bus that was scrapped years ago

Let’s start with a concrete example. This is adapted from the Go 1.26 release notes, and it’s the kind of bug that makes it into production more often than anyone wants to admit:

type result struct {
    res workResult
    err error
}

func processWorkItems(ws []workItem) ([]workResult, error) {
    // Process work items in parallel, aggregating results in ch.
    ch := make(chan result)
    for _, w := range ws {
        go func(w workItem) {
            res, err := processWorkItem(w)
            ch <- result{res, err}
        }(w)
    }

    // Collect the results from ch, or return an error if one is found.
    var results []workResult
    for range len(ws) {
        r := <-ch
        if r.err != nil {
            // This early return may cause goroutine leaks.
            return nil, r.err
        }
        results = append(results, r.res)
    }
    return results, nil
}

Looks reasonable, right? Spin up some workers, collect their results, bail out early if any of them fail. Standard concurrent programming stuff.

Now let’s walk through what actually happens. Say you have 10 work items. You spawn 10 goroutines, each one racing to process its item and send the result back through ch. The first 3 finish quickly and send their results. The 4th one hits an error and sends that back. Your main goroutine receives it and immediately returns with the error.

What happens to the other 6 goroutines? They’re still running. They finish processing their work items. They try to send their results to ch. And they block. Forever.

Why? Because ch is unbuffered, and the receiver (your main goroutine) just left. The channel becomes unreachable—there’s literally nobody left in the program who has a reference to it. But those 6 goroutines don’t know that. They’re stuck in a send operation, waiting for a receiver that will never come. They’re waiting for a bus that was scrapped years ago.

It’s not that hard. right?

“Okay”, you might be thinking:

but that’s just a bug in the code. fix the code, problem solved

Sure.

Except goroutine leaks aren’t like other bugs. They’re insidious.

They don’t crash your program. Unlike a nil pointer dereference or an out-of-bounds access, a leaked goroutine just… sits there. Your program keeps running. Everything looks fine. Until it doesn’t.

They accumulate gradually. In production, this leak might happen once every thousand requests. At first, it’s invisible. You’ve got plenty of memory. But over hours or days, those leaked goroutines pile up. Each one holds onto its stack (minimum 2KB, can grow to megabytes), any variables it has references to, and any resources it opened. Eventually, your program starts getting OOM-killed, or it becomes so slow from GC pressure that you have to restart it.

They’re hard to debug. When you finally notice something’s wrong and grab a goroutine profile, what do you see? “10,000 goroutines blocked on channel send.” Great. But which ones are legitimate (waiting for work, waiting on slow I/O, part of normal operation) and which ones are leaked? You can’t tell. A waiting goroutine looks the same whether it’s going to wake up in 10 milliseconds or never.

This pattern shows up everywhere:

  • Worker pools that forget to close their input channel.
  • Context cancellation that orphans the sender.
  • Error handling that returns early without cleaning up.
  • Timeouts that fire while goroutines are still processing.

The Go team validated this feature against 66 real bugs from production codebases, collected in the GoKer test suite. CockroachDB alone had 18 instances of this class of bug. Kubernetes had 14. Moby had 11. etcd, gRPC, Istio. Every major Go project has shipped code that leaks goroutines.

What makes detection hard

The fundamental problem from the runtime’s perspective: A waiting goroutine looks healthy. Consider these three goroutines:

  1. Waiting on a channel for the next work item (legitimate)
  2. Waiting on a slow database query (legitimate)
  3. Waiting on a channel that nobody will ever send to (LEAK!)

They all have the same status: _Gwaiting. They all look the same in a goroutine profile. Traditional timeout-based leak detectors would flag #2 as a leak if the database query takes too long, giving you false positives. And they’d miss #3 entirely if you don’t wait long enough.

What you really need to know is: CAN this goroutine ever wake up?

For #1 and #2, the answer is yes. Someone has a reference to the channel or the network connection, and they could send/complete at any time. For #3, the answer is no. The channel is unreachable from any running code. Nobody can send to it. The goroutine is stuck forever.

Ask the garbage collector

Here’s where it gets interesting. Go already has something that’s really good at determining reachability: the garbage collector.

The GC’s whole job is to figure out which objects are still reachable from your program and which ones aren’t. It knows which channels have references from running goroutines, which ones are reachable from global variables, which ones are sitting in stack frames. It has to know this, or it can’t do its job.

So the question becomes: what if we asked a different question?

Not “is this goroutine waiting?” but “CAN this goroutine ever wake up?”

And to answer that: check if the channel (or mutex, or WaitGroup, or whatever) the goroutine is blocked on is reachable from the rest of your program. If nobody has a reference to it, if the GC would mark it as garbage, then the goroutine is leaked. It’s semantically deadlocked, stuck forever waiting on something that can’t possibly happen.

“And conveniently,” as the runtime team probably thought, “we already have a garbage collector that knows exactly which objects are reachable.”

How to ask the GC nicely

Go 1.26’s experimental goroutine leak detector builds on this insight. When you request a goroutineleak profile, the runtime triggers a special GC cycle that performs reachability analysis specifically for leak detection.

Here’s how you use it:

import "runtime/pprof"

prof := pprof.Lookup("goroutineleak")
prof.WriteTo(os.Stdout, 2)

Or if you’re already serving pprof over HTTP:

curl http://localhost:6060/debug/pprof/goroutineleak

The output looks similar to a regular goroutine profile, except leaked goroutines are annotated with (leaked):

goroutine 42 [chan send (leaked)]:
main.processWorkItems.func1()
    /path/to/file.go:25 +0x45
created by main.processWorkItems
    /path/to/file.go:20 +0x123

That (leaked) annotation is the key.

The regular goroutine profile tells you HOW MANY goroutines are waiting.

The leak profile tells you WHICH ONES are never waking up.

For the processWorkItems bug we looked at earlier, a regular goroutine profile would show:

6 goroutines blocked on channel send

Could be fine, could be a leak, who knows? But the leak profile shows:

goroutine 10 [chan send (leaked)]:
goroutine 11 [chan send (leaked)]:
goroutine 12 [chan send (leaked)]:
goroutine 13 [chan send (leaked)]:
goroutine 14 [chan send (leaked)]:
goroutine 15 [chan send (leaked)]:

Now we’re talking. Six goroutines, all leaked, all from the same code path.

That’s actionable information.

What this unlocks

The implications are bigger than just finding bugs. This changes how you can think about testing concurrent code.

Imagine running your integration tests with leak detection enabled. Any test that leaks a goroutine fails immediately.

No more

we’ll notice it in production when memory usage climbs

No more

let’s add some time.Sleep calls and hope the goroutines finish

You get deterministic detection of a class of bugs that used to be nearly impossible to catch in tests.

Or picture this: you’re debugging a production incident where memory usage is climbing. You grab a leak profile and see exactly which goroutines are stuck and where they were created. Not “we have a goroutine leak somewhere”, but “we’re leaking goroutines from this specific HTTP handler when requests time out.”

The leak detector doesn’t replace good concurrent programming practices. It’s not a magic wand that makes channel bugs disappear. What it does do is surface a class of bugs that previously required either dumb luck or painful debugging sessions with heap dumps and scattered time.Sleep calls.
