Migrating a Zig UUID library to 0.16

17 April 2026 / 6 min read
  • zig
  • systems programming
  • uuid

Zig 0.16 dropped, and it broke everything. Not in the “oops, a function got renamed” way. More like “the foundational assumption of how your code talks to the operating system just changed.” In my experience, that’s exactly the kind of breakage that teaches you the most about a language’s direction.

I maintain a small RFC 9562 UUID library in Zig. Zero allocations, no panics on clock issues, strict compliance. The kind of library where you obsess over bit-packing and feel smug when paired assertions catch a hypothetical off-by-one in a nibble shift. The migration to 0.16 took about an hour. Here’s what happened.

Everything wants Io now

My UUID generators used to look like this:

pub fn v4() Self {
    var uuid: Self = undefined;
    std.crypto.random.bytes(&uuid.bytes);
    // stamp version and variant...
    return uuid;
}

Clean, simple, zero parameters. In 0.16, std.crypto.random doesn’t exist anymore. This is what it looks like now:

pub fn v4(io: std.Io) Self {
    var uuid: Self = undefined;
    io.random(&uuid.bytes);
    // stamp version and variant...
    return uuid;
}

The headline change in 0.16 is std.Io. The standard library used to give you global access to things like std.crypto.random, std.time.nanoTimestamp(), and std.process.args(). Just call them. They reach into the OS and hand you what you need.

That’s gone.

Now these operations go through an Io interface, a vtable-based abstraction that gets threaded through your program starting from main. Your main function signature changes from pub fn main() void to pub fn main(init: std.process.Init) void, and init.io is how you talk to the world.

For an application, this is straightforward. For my library, the question was: where does Io come from? The answer turned out to be simple. Pass it as a parameter to every function that has side effects.

This affected v1, v4, v6, and v7. The deterministic generators (v3, v5, v8) and all the FromFields variants stayed untouched. They’re pure functions of their inputs, no side effects, no Io. The boundary landed exactly where it should.

Keeping the comptime clock injection alive

The library uses a pattern I like: UuidImpl is a generic function parameterized by clock sources. Production code uses real clocks, tests inject fakes to exercise clock stall paths without actually waiting for the system clock.

pub const Uuid = UuidImpl(defaultNanoTimestamp, defaultMilliTimestamp);

The trick was updating the clock function signatures. They went from fn() i128 and fn() i64 to fn(Io) i96 and fn(Io) i64. The i128 to i96 change mirrors Io.Timestamp.toNanoseconds() returning i96 instead of the old i128. Plenty of headroom for UUID timestamps either way.

The clock stall tests survived almost unchanged. The fake clocks just ignore the Io parameter:

const FrozenUuid = UuidImpl(defaultNanoTimestamp, struct {
    fn clock(_: Io) i64 { return 1000; }
}.clock);

Same pattern, one extra underscore. I’ll take it.

The smaller breaks

std.meta.intToEnum got removed. My getVersion() used it to safely convert a 4-bit nibble to a Version enum. The fix was a plain switch, which honestly reads better anyway. No need for reflection when you have eight cases.

std.time.nanoTimestamp() and std.time.milliTimestamp() are gone entirely. Replaced by Io.Timestamp.now(io, .real) followed by .toNanoseconds() or .toMilliseconds(). More explicit about which clock you’re reading, which is probably a good thing when you’re implementing RFC-specified timestamp semantics.

The format function signature for std.fmt integration changed from writer: anytype to writer: *std.Io.Writer. Concrete type instead of duck typing. The {f} specifier still dispatches to it.

On the CLI side, std.process.args() became init.minimal.args.iterate(), and std.fs.File.stdout().writerStreaming(&buf) became std.Io.File.stdout().writerStreaming(io, &buf). The error.BrokenPipe you’d catch when piped to head is now just error.WriteFailed, since the writer abstraction flattened the error set.

Taken together, these changes all push in the same direction: concrete types, explicit dependencies, no more magic globals.

So where’s the allocation?

If you’ve been paying attention, you might have a question. A vtable-based interface threaded through every side-effect call? That sounds like it costs something. And in most languages, it does.

In Go, an interface can box a value onto the heap. In Java, virtual dispatch implies a heap-allocated object behind it. In C++, you typically new your polymorphic objects. Years of these languages trained us to associate “interface” with “allocation.”

Zig decouples the two.

std.Io is just two fields:

userdata: ?*anyopaque,
vtable: *const VTable,

The vtable is a static table of function pointers living in .rodata, known at compile time. The userdata points to whatever backing implementation the caller provides. Neither is heap-allocated. The whole thing is 16 bytes on the stack, passed by value through the call chain.

When I call io.random(&uuid.bytes), what happens is one indirect function call. The vtable dispatches to the real implementation, which fills my stack-allocated buffer. No boxing, no allocation, no hidden malloc behind the abstraction.

Zig could have gone the comptime generics route for this, which would eliminate even the indirect call. But Io needs to cross module and ABI boundaries. A vtable is the right tool when the implementation isn’t known at compile time in the consuming code. The cost is one pointer chase per operation. For a UUID generator that’s already doing a syscall to read random bytes from the OS, that pointer chase is noise.

My library is still zero-allocation. The Io parameter changed the API surface, not the memory model.

The design signal

The Io pattern is the right call for Zig. It makes side effects visible in function signatures, which is exactly what you want in a systems language that cares about testability and determinism. My clock stall tests are a perfect example: they work because I had designed the library around injectable dependencies from the start. Zig 0.16 just made that the standard way to do it instead of a clever library trick.

The migration cost was real but bounded. Every std.crypto.random.bytes() call became io.random(). Every timestamp fetch gained an io parameter. The ripple went from my generators, through the internal helpers, up to the public API. Anyone using the library will need to update their call sites.

What strikes me is that this is exactly the kind of breaking change that gets easier the smaller your library is. An UUID library with zero allocations and explicit dependencies? One hour. A large application with globals scattered through 50 files? That’s going to be a different conversation.

All tests pass. The CLI works. The bit-packing is still correct. On to the next thing.
