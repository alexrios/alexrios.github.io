How a routine Pop!_OS update turned my Monday into a kernel debugging session

22 December 2025 / 4 min read
  • linux
  • pop-os
  • troubleshooting

There’s a particular kind of dread that hits when you see your screen filled with BUG: soft lockup messages instead of your desktop. It’s the Linux equivalent of watching your car’s check engine light turn into the entire dashboard going red.

The setup

I did what any reasonable person does when their system politely asks to update: I clicked “Yes” and went to put some food for the cats. Pop!_OS has been solid for years. What could go wrong? (the famous last words)

The answer, as it turns out, is kernel 6.17.4 combined with NVIDIA drivers. The update pulled in a shiny new kernel that apparently hadn’t been properly introduced to my GPU (RTX 4090 btw).

The symptoms

After the reboot, my monitor displayed what looked like a kernel panic’s chatty cousin:

watchdog: BUG: soft lockup - CPU#10 stuck for 417s
watchdog: BUG: soft lockup - CPU#8 stuck for 257s
watchdog: BUG: soft lockup - CPU#14 stuck for 440s

Multiple CPUs, all stuck. The culprits? systemd-udevd and modprobe. The kernel was trying to load drivers and getting nowhere. Think of it as a traffic jam at a roundabout where everyone’s waiting for everyone else to move first.

After a hard reboot and another attempt, it escalated from soft lockup to hard lockup. The CPU wasn’t just stuck in a loop anymore; it was completely frozen. Progress, of sorts.

The recovery ritual

Pop!_OS has a recovery mode, which saved me from having to dig out a live USB. Booting into recovery gave me a shell, but here’s where things got interesting.

My first instinct was to reinstall the NVIDIA drivers. Running apt install --reinstall system76-driver-nvidia seemed to work, but when I tried to rebuild the initramfs, I got a cascade of Python tracebacks complaining about /proc/cmdline not existing.

The problem? I was running commands directly in the recovery environment instead of properly chrooting into my installed system (my bad). The recovery environment is its own minimal Linux; it doesn’t have access to all the virtual filesystems that a running system needs.

The actual fix

The proper recovery workflow looks like this:

# Mount the root partition
sudo mount /dev/nvme0n1p3 /mnt

# Mount the virtual filesystems the system needs
sudo mount --bind /dev /mnt/dev
sudo mount --bind /dev/pts /mnt/dev/pts
sudo mount --bind /proc /mnt/proc
sudo mount --bind /sys /mnt/sys

# Mount EFI partition
sudo mount /dev/nvme0n1p1 /mnt/boot/efi

# Chroot into the system
sudo chroot /mnt

Now I’m operating inside my installed system, not alongside it.

From there, the fix was straightforward: don’t use the broken kernel. I had two kernels available:

$ ls /boot/vmlinuz-*
/boot/vmlinuz-6.12.10-76061203-generic  /boot/vmlinuz-6.17.4-76061704-generic

The 6.12 kernel worked fine before the update. So I set it as the default boot kernel:

sudo kernelstub -k /boot/vmlinuz-6.12.10-76061203-generic \
                -i /boot/initrd.img-6.12.10-76061203-generic --verbose

The dependency hell bonus round

Of course, apt had opinions about this. Trying to remove the broken kernel triggered dependency resolution failures because pop-server requires some kernel to be installed. The package manager was trying to be helpful by reinstalling the exact kernel I was trying to escape.

The solution: let the broken kernel exist but tell apt to stop touching it:

sudo apt-mark hold linux-image-6.17.4-76061704-generic \
                   linux-headers-6.17.4-76061704-generic

Now the system boots on 6.12.10, apt is happy because technically the 6.17.4 packages are “installed,” and I can wait for Pop!_OS to ship a fixed kernel.

RECAP

1. Kernel updates are not routine updates. They touch everything. New kernel + proprietary GPU drivers = potential for exciting failures.

2. Know your recovery workflow before you need it. The difference between “mount and chroot” and “run commands in recovery mode” is the difference between fixing your system and creating new problems.

3. Hold packages when rolling back. Package managers want to “fix” your system by reinstalling what you removed. Tell them not to.

4. Keep the previous kernel around. Pop!_OS does this by default, which saved me. If your distro doesn’t, configure it to keep at least one previous kernel.

5. NVIDIA + Linux + bleeding edge = choose two. This isn’t news to anyone who’s been around Linux for a while, but it’s a good reminder that staying one kernel version behind on systems with proprietary drivers is often the pragmatic choice.

The current state

System’s running fine on kernel 6.12.10. When Pop!_OS releases a fixed 6.17.x (or 6.18), I’ll unhold the packages and try again:

sudo apt-mark unhold linux-image-6.17.4-76061704-generic \
                     linux-headers-6.17.4-76061704-generic
sudo apt update && sudo apt upgrade

Until then, I have a working system and a renewed appreciation for the boring reliability of not being on the latest kernel.
