Building a UUID library in Zig over a weekend

28 March 2026 / 6 min read
  • zig
  • uuid
  • systems
  • devlog

Most UUID libraries get v4 right and ignore everything else. I built one that doesn’t. It implements all versions from RFC 9562 (except v2), allocates nothing on the heap, and ships as both a library and a CLI.

Therefore, welcome again. This is the design journal.

Why another UUID library

Most UUID libraries treat the spec as “generate 128 random bits, stamp a version nibble (4-bit version field).” That works for v4. It doesn’t work for v1, v6, or v7, versions where timestamps, clock sequences, and monotonicity guarantees matter. I checked what existed in the Zig ecosystem and found nothing that covered the full spec.

The goal was a library that:

  • Follows the actual RFC 9562, not a simplified subset.
  • Never touches the heap (16 bytes on the stack, always).
  • Makes clock-related bugs impossible to hide.
  • Compiles to something I can embed anywhere Zig runs.

Zig was the vehicle because its compile-time features let me do things that would require runtime indirection in other languages.

The constraint: zero heap allocations

A UUID is 16 bytes. The canonical string representation is 36 bytes. Both fit on the stack. So the rule was simple: no allocator parameter, ever.

pub const Uuid = struct {
    bytes: [16]u8,

    pub const nil: Uuid = .{ .bytes = .{0} ** 16 };
    pub const max: Uuid = .{ .bytes = .{0xff} ** 16 };

    pub fn toStr(self: Uuid) [36]u8 {
        // Stack-allocated, returned by value
        var out: [36]u8 = undefined;
        // ...formatting logic...
        return out;
    }
};

This has a ripple effect on the entire API. Parsing returns a Uuid by value. Formatting returns a [36]u8 by value. No string ownership, no cleanup, no allocator threading. The caller’s stack frame is the only memory involved. If your data fits in a fixed-size buffer, skipping the allocator simplifies every call site downstream.

The namespace constants (namespace_dns, namespace_url, etc.) are parsed at comptime:

pub const namespace_dns: Uuid = parse("6ba7b810-9dad-11d1-80b4-00c04fd430c8") catch unreachable;

The catch unreachable is safe here because the input is a literal. If parsing somehow failed, the build would break, not the running program.

API surface decisions

The library covers all RFC 9562 versions except v2 (DCE Security, rarely used):

VersionInputUse case
v1System clock + nodeLegacy compatibility
v3Namespace + name (MD5)Deterministic, deprecated
v4RandomGeneral purpose
v5Namespace + name (SHA-1)Deterministic
v6System clock + nodev1 but sortable
v7Unix timestamp + counterSortable, modern
v8Caller-provided fieldsCustom

Every generator follows the same contract:

  1. Fill the 16-byte buffer with the appropriate data.
  2. Stamp version and variant bits with setVersion / setVariant.
  3. Verify the result with an independent read-back (getVersion / getVariant).

That third step (the paired assertion) turned out to be one of the most valuable patterns in the codebase. It catches bit-scatter bugs through a completely independent code path. Paired assertions are cheap insurance: use them anywhere bit manipulation touches shared state.

Per-thread monotonicity

v1, v6, and v7 must produce monotonically increasing UUIDs. The RFC allows this to be per-thread, and that’s what I chose.

v1 and v6 share a Gregorian clock state:

const GregorianState = struct {
    last_timestamp: u60 = 0,
    clock_seq: u14 = 0,
    node: [6]u8 = .{0} ** 6,
    initialized: bool = false,
};

threadlocal var gregorian_state: GregorianState = .{};

v7 has its own state:

const V7State = struct {
    last_ms: i64 = 0,
    counter: u12 = 0,
    initialized: bool = false,
};

threadlocal var v7_state: V7State = .{};

The threadlocal keyword is doing the heavy lifting. Each thread gets independent counters. No locks, no atomics, no contention. The trade-off is explicit: UUIDs from different threads don’t sort relative to each other. That’s fine. The RFC permits this.

When the clock goes backwards (NTP correction, VM migration), the counter increments instead of re-randomizing. This preserves monotonicity during brief regressions. If the counter overflows, we spin-wait for the clock to advance.

Clock overflow: spin-wait to error.ClockStall

The most interesting design decision was what to do when a generator runs out of counter space. For v7, that’s 4096 UUIDs in one millisecond. For v1/v6, it’s 16384 in one 100-nanosecond tick.

Options considered:

  1. Block until the clock advances. Simple, but unbounded.
  2. Return an error. Forces the caller to handle it, but disruptive.
  3. Spin-wait with a timeout, then error. Bounded wait, graceful degradation.

Option 3 won:

const max_spin_iterations: u32 = 2_000_000;

// In v7():
if (v7_state.counter == std.math.maxInt(u12)) {
    for (0..max_spin_iterations) |_| {
        if (milliTimestampFn() > v7_state.last_ms) break;
        std.atomic.spinLoopHint();
    } else {
        return error.ClockStall;
    }
    continue; // retry with new millisecond
}

spinLoopHint() maps to x86 PAUSE, whose latency varies widely by microarchitecture: ~10 cycles on older Intel, ~140 cycles since Skylake, ~65 on AMD Zen2. At 2 million iterations, the actual wait ranges from roughly 7ms to 90ms depending on hardware, comfortably above the 1ms clock tick needed for the millisecond counter to advance. If it doesn’t advance in that window, something is genuinely wrong (frozen VM, broken clock source), and the caller needs to know.

The error.ClockStall is a real error, not a panic. The caller decides what to do: retry, fall back to v4, log and continue.

Shipping: from library to CLI

The project is structured as a Zig module (the library) plus an executable (the CLI):

// build.zig
const uuid_mod = b.addModule("uuid", .{
    .root_source_file = b.path("src/uuid.zig"),
});

const exe = b.addExecutable(.{
    .name = "uuid",
    .root_module = b.createModule(.{
        .root_source_file = b.path("src/main.zig"),
        .imports = &.{
            .{ .name = "uuid", .module = uuid_mod },
        },
    }),
});

The CLI is minimal. Two commands: generate and parse.

$ uuid generate v7
019614a2-3b80-7cc3-98c4-dc0c0c07398f

$ uuid parse 019614a2-3b80-7cc3-98c4-dc0c0c07398f
UUID:    019614a2-3b80-7cc3-98c4-dc0c0c07398f
Version: 7 (Unix epoch time-based)
Variant: RFC 9562

One detail worth mentioning: BrokenPipe handling. When piped to head or grep, the reader closes the pipe early. Without handling this, the process exits with a non-zero status and a confusing error. The fix is two lines:

pub fn main() void {
    run() catch |err| {
        if (err == error.BrokenPipe) return;
        std.process.exit(1);
    };
}

For distribution, I used GoReleaser. It’s a Go tool, but it handles cross-compilation archives, checksums, and Homebrew tap updates regardless of language. The CI pipeline builds ReleaseSafe (optimized with runtime safety checks) binaries for Linux (x86_64, aarch64) and macOS (x86_64, aarch64). Correctness matters more than nanoseconds here. There’s no meaningful performance to gain from ReleaseFast, but there’s real value in keeping the safety net.

What I’d do differently

More explicit about the per-thread boundary. The documentation says monotonicity is per-thread, but the API doesn’t enforce it. A future version could return a ThreadUuidGenerator that’s explicitly non-Send, making the boundary visible at the type level.

Benchmark the spin-wait budget. The 2-million-iteration constant works on my hardware. It might be too aggressive on embedded targets or too conservative on high-frequency systems. Making it configurable (via comptime parameter) would be a small change with real value.

Consider v2. RFC 9562 reserves v2 (DCE Security UUIDs) but delegates its definition to a separate spec. I skipped it because it’s rarely used and requires OS-level UID/GID access. Including it for completeness wouldn’t be hard, but it would be the first version that needs platform-specific code beyond the clock.

The common thread: make implicit contracts explicit at the type level, and make configuration visible at compile time. These are cheap in Zig. I should have done them from the start. But my time budget for this side project was two days, and they’re gone.

Good enough to ship, clear enough to revisit.
