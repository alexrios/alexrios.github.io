I was reading the Go source code and found a bug

16 May 2026 / 13 min read
  • go
  • runtime
  • cpu
  • simd
  • gc
  • internals

Revisions: v1 (2026-05-16) initial post.

While prepping my GopherCon Europe talk, I noticed something in the Go source that looked like a bug.

It’s a one-liner. The kind that sits at the intersection of Intel’s CPUID table, Go’s feature-detection package, and the SIMD codepaths the Green Tea GC team wired up in Go 1.26. None of those things look at each other very often. So when the wiring is wrong, nobody notices for a year. Then someone preparing a conference talk decides to follow the AVX-512 detection path all the way down to the metal.

This post is the full story: the line that caught my eye, why it’s wrong, how I proved it on my desk, what the fix is, and why the user-visible impact is both smaller and bigger than you’d think at the same time.

Where I was

The talk is about how Green Tea’s SIMD acceleration actually reaches hardware, so I was rereading the gate that decides whether a span can take the SIMD scan kernel. It’s the same gate I wrote about in the Green Tea series:

// src/internal/runtime/gc/scan/scan_amd64.go
var avx512ScanPackedReqsMet = cpu.X86.HasAVX512VL &&
    cpu.X86.HasAVX512BW &&
    cpu.X86.HasGFNI &&
    cpu.X86.HasAVX512BITALG &&
    cpu.X86.HasAVX512VBMI &&
    cpu.X86.HasPOPCNT

HasGFNI is in there because the scan kernel uses VGF2P8AFFINEQB, Intel’s Galois-Field affine transform instruction, to expand object-level mark bits into word-level scan bits with a single matrix multiply over GF(2). I wrote about that trick in detail in Part 4 of the Green Tea series. Cool instruction. Cool reuse.

To understand the GC’s hardware requirements I clicked through to internal/cpu/cpu_x86.go to see where HasGFNI is set. And then I sat there for a minute.

The line

Here is roughly what the file looked like before the fix (Go 1.27 tip, pre-patch):

X86.HasAVX512F = isSet(ebx7, cpuid_AVX512F) && osSupportsAVX512
if X86.HasAVX512F {
    X86.HasAVX512CD = isSet(ebx7, cpuid_AVX512CD)
    X86.HasAVX512BW = isSet(ebx7, cpuid_AVX512BW)
    X86.HasAVX512DQ = isSet(ebx7, cpuid_AVX512DQ)
    X86.HasAVX512VL = isSet(ebx7, cpuid_AVX512VL)
    X86.HasAVX512GFNI = isSet(ecx7, cpuid_AVX512GFNI)
    // ... lots of AVX-512-specific flags ...
    X86.HasGFNI = isSet(ecx7, cpuid_GFNI)   //  <- this line
    X86.HasAVX512BITALG = isSet(ecx7, cpuid_AVX512_BITALG)
}

X86.HasGFNI is being assigned inside if X86.HasAVX512F. Which means: on any CPU where AVX-512F is not present, HasGFNI stays at its zero value (false), regardless of what the actual GFNI bit in CPUID reports.

That would be fine if GFNI required AVX-512. It does not.

A one-paragraph hardware refresher

GFNI is a small, standalone ISA extension Intel introduced on Ice Lake server (2019) and backported to a whole pile of newer cores. It ships in three encodings:

  • Legacy SSE: GF2P8AFFINEQB on XMM registers (128-bit), no VEX prefix, no OS-level vector state required beyond what SSE2 already needs.
  • VEX-128 and VEX-256: VGF2P8AFFINEQB on XMM or YMM, requires AVX (which in turn requires XSAVE plus OS-level YMM save/restore).
  • EVEX (128/256/512): VGF2P8AFFINEQB on XMM/YMM/ZMM, requires AVX-512F.

Only the third encoding needs AVX-512. The Intel ISA Extensions reference is explicit about this: GFNI was deliberately defined with non-AVX-512 encodings so that Intel could ship the instruction on cores that never had, and never will have, AVX-512. That includes the entire Atom line and, since 2021, the entire consumer desktop/laptop line, because Intel fused AVX-512 off on Alder Lake and everything after to keep the P-cores and E-cores at the same ISA level.

In CPUID terms there is one bit for the feature itself:

CPUID.(EAX=7, ECX=0).ECX[8]  =  GFNI

and a separate bit for the EVEX form:

CPUID.(EAX=7, ECX=0).ECX[8]   tells you GFNI exists
plus AVX512F                  tells you the EVEX-512 form is usable

Go has two distinct flags to match: X86.HasGFNI (the feature exists, VEX form usable on AVX hardware) and X86.HasAVX512GFNI (the EVEX form is usable). The bug was that HasGFNI was being set in lockstep with HasAVX512GFNI instead of independently.

Who actually has GFNI without AVX-512?

A large and growing list. Intel Alder Lake (12th gen Core, 2021) shipped with AVX-512 silicon present on the P-cores; early boards could enable it only by disabling the E-cores entirely. Intel removed the option in microcode soon after, and on later steppings the feature is fused off in silicon. GFNI shipped intact. Raptor Lake (13th/14th gen, 2022–2023), Meteor Lake, Arrow Lake, and Lunar Lake (2024+) all follow the same shape: no AVX-512, full GFNI. The Atom-derived parts from Tremont onward (Jasper Lake, Elkhart Lake, Gracemont, Crestmont) also have it. That family covers the N-series (N100/N200/N300) that shows up in basically every cheap mini-PC, NAS, and edge-compute box on the market right now.

That’s a lot of hardware. A lot of hardware on which Go was reporting HasGFNI = false despite the CPU happily executing VGF2P8AFFINEQB on YMM.

Reproducing it on my desk

My main workstation is an i9-14900K. Raptor Lake refresh. Exactly the buggy scenario.

A quick grep -m1 ^flags /proc/cpuinfo confirms:

flags : ... avx avx2 ... gfni vaes vpclmulqdq ...

gfni present. No avx512* flag anywhere.

I wrote a small regression test that reads CPUID leaf 7 directly and compares it against the value Go’s internal/cpu package reports:

func TestX86ifCPUIDGFNIhasGFNI(t *testing.T) {
    if !X86.HasAVX {
        t.Skip("requires AVX (and thus YMM OS support)")
    }
    maxID, _, _, _ := Cpuid(0, 0)
    if maxID < 7 {
        t.Skip("CPUID leaf 7 unavailable")
    }
    _, _, ecx7, _ := Cpuid(7, 0)
    const gfniBit = 1 << 8
    if ecx7&gfniBit != 0 && !X86.HasGFNI {
        t.Fatalf("HasGFNI expected true when CPUID leaf 7 ECX bit 8 is set, got false")
    }
}

This test sits on the branch I’m preparing. Running it against an unpatched tree on the 14900K, here’s the failure:

=== RUN   TestX86ifCPUIDGFNIhasGFNI
    cpu_x86_test.go: HasGFNI expected true when CPUID leaf 7 ECX bit 8 is set, got false
--- FAIL: TestX86ifCPUIDGFNIhasGFNI (0.00s)

Hardware says yes. Go says no.

The fix

The change is short. Move HasGFNI out of the AVX-512 block and gate it on HasAVX:

// GFNI is available in VEX-encoded form on every CPU that reports the
// GFNI bit (CPUID leaf 7 ECX bit 8) and supports YMM, regardless of
// AVX-512. Gate on HasAVX (which requires OSXSAVE + YMM OS support) so
// callers can safely emit VEX-256 VGF2P8AFFINEQB on YMM. The
// EVEX-encoded form additionally requires AVX-512F; see HasAVX512GFNI
// below.
X86.HasGFNI = isSet(ecx7, cpuid_GFNI) && X86.HasAVX

Three things to notice about this line:

  1. The pattern is not new. Three lines above it, HasVAES is gated exactly the same way: isSet(ecx7, cpuid_VAES) && X86.HasAVX. VAES has the same SSE/VEX/EVEX story as GFNI. The fix is making HasGFNI follow the convention that was already there for its sibling extension.

  2. HasAVX does the heavy lifting. Why gate on HasAVX and not directly on the AVX CPUID bit? Because HasAVX is the already-cooked value: it’s true only when the CPU has the AVX bit and OSXSAVE is enabled and the OS has indicated (via XGETBV / XCR0) that it will save and restore YMM state across context switches. That’s the actual precondition for VEX-encoded instructions to be safe. Anything weaker risks #UD on hosts where the OS doesn’t track YMM.

  3. HasAVX512GFNI is untouched. The EVEX path was already correctly gated by the surrounding if X86.HasAVX512F block, and it still is. The two flags now report what they’re supposed to report independently.

To defend against future regressions, the same patch adds an invariant test:

func TestX86ifGFNIhasAVX(t *testing.T) {
    if X86.HasGFNI && !X86.HasAVX {
        t.Fatalf("HasAVX expected true when HasGFNI is true, got false")
    }
}

There’s a matching one for the EVEX side: HasAVX512GFNI must imply HasGFNI. These tests are cheap insurance. They won’t catch every misencoding, but they will catch any future refactor that accidentally re-nests HasGFNI back inside the AVX-512 block.

A note on the vendored copy

The fix lives in src/internal/cpu/cpu_x86.go. There is a separate copy of the same logic at src/vendor/golang.org/x/sys/cpu/cpu_x86.go, which Go vendors from the upstream golang.org/x/sys/cpu module for use by tooling. That copy has the exact same bug and is not fixed by the patch I’m preparing. Vendored copies are refreshed from upstream, not edited in place. A proper follow-up is sending the same change upstream to golang.org/x/sys/cpu. Anyone using that library in their own code today on Alder Lake is getting the same wrong answer, just from a different file.

You can spot the vendored mismatch experimentally: run any Go test with GODEBUG=cpu.gfni=off and you’ll see this warning on stderr:

GODEBUG sys/cpu: unknown cpu feature "gfni"

That’s the vendored package complaining that it doesn’t know what gfni is. Harmless, but visible.

A GODEBUG detour

Once HasGFNI becomes correctly reported on millions of CPUs that previously had it forced to false, you suddenly want an off switch. Before this fix, the only way to suppress GFNI on a non-AVX-512 host was the sledgehammer GODEBUG=cpu.avx=off, which disables every AVX feature in the runtime. That’s a terrible debugging tool. If you suspect a GFNI codegen bug, you should be able to disable just GFNI.

So the patch also adds an entry to the options table:

options = []option{
    {Name: "adx", Feature: &X86.HasADX},
    {Name: "aes", Feature: &X86.HasAES},
    {Name: "erms", Feature: &X86.HasERMS},
    {Name: "fsrm", Feature: &X86.HasFSRM},
    {Name: "gfni", Feature: &X86.HasGFNI},   // <- new
    {Name: "pclmulqdq", Feature: &X86.HasPCLMULQDQ},
    // ...
}

Now GODEBUG=cpu.gfni=off clears X86.HasGFNI after init. There’s a TestDisableGFNI test that spawns a child process with that variable set and asserts the flag flips to false.

This is the right ergonomics for the user. It is also where the patch grew an interesting wrinkle.

The wrinkle: asymmetric kill switches

HasGFNI is now in the options table. HasAVX512GFNI is not in the options table. (Most AVX-512 sub-features aren’t.) Which means on a CPU that has both (Tiger Lake, Ice Lake, Sapphire Rapids, AMD Zen 4) running with GODEBUG=cpu.gfni=off will produce this state:

X86.HasGFNI       == false
X86.HasAVX512GFNI == true

That breaks the invariant the patch otherwise wants to assert: “if HasAVX512GFNI is true, then HasGFNI is true.” So the test needs a skip guard:

func TestX86ifAVX512GFNIhasGFNI(t *testing.T) {
    if godebug.New("#cpu.gfni").Value() == "off" ||
        godebug.New("#cpu.all").Value() == "off" {
        t.Skip("GODEBUG=cpu.gfni=off set")
    }
    if X86.HasAVX512GFNI && !X86.HasGFNI {
        t.Fatalf("HasGFNI expected true when HasAVX512GFNI is true, got false")
    }
}

The two godebug.New("#...") calls cover the direct case (cpu.gfni=off) and the global case (cpu.all=off), because processOptions honors both.

Is the skip guard satisfying? Honestly, no. It’s a test-side band-aid for a small production-side inconsistency. The cleaner fix would be to derive X86.HasAVX512GFNI = isSet(ecx7, cpuid_AVX512GFNI) && X86.HasGFNI, so the EVEX flag tracks the VEX flag by construction and the kill switch suppresses both. But that’s a larger semantic change to the public-ish internal/cpu surface, and I kept the diff minimal and the surface unchanged. The skip guard is the smallest local fix.

It is worth noting why the symmetric tests for AVX-512BW/VL/etc. don’t need this guard: those features all have their parent avx512f in the options table when GOAMD64 < 4, so a global kill switch flips parent and child in lockstep. The GFNI pair is the unique case where the child has a kill switch but the parent doesn’t.

What does this actually buy users?

This is the question I want to answer honestly. The temptation with a patch like this is to oversell it as “your Go programs are now faster on Alder Lake.” They are not. Here’s the accurate version.

As of Go 1.27 tip (May 2026), the only consumer of X86.HasGFNI in the Go standard library is the Green Tea scan kernel. That consumer ANDs HasGFNI with five AVX-512 flags (HasAVX512VL, HasAVX512BW, HasAVX512BITALG, HasAVX512VBMI, HasPOPCNT). On Alder Lake or my 14900K, those AVX-512 flags are all false, so the conjunction is false regardless of whether HasGFNI is fixed. The scan kernel still falls back to the scalar path. Wall-clock GC time on a Raptor Lake desktop does not change.

So if you read this far hoping I’d tell you that a Go upgrade is about to speed up your build server: it isn’t. Not from this patch alone.

What the patch does change is what becomes possible going forward. Specifically:

  • A future VEX-256 variant of the Green Tea scan kernel, using VGF2P8AFFINEQB on YMM instead of ZMM, could ship and actually fire on the huge installed base of non-AVX-512 silicon. Without this fix it would never reach hardware because HasGFNI would lie. The 256-bit kernel wouldn’t be as fast as the 512-bit one, but it would still beat scalar on the spans where SIMD setup pays off.
  • Other stdlib hot paths that could plausibly use GFNI (encoding/base64, encoding/hex, UTF-8 validation, GF(2⁸)-based Reed-Solomon for erasure coding, certain CRC variants) now have an accurate detection bit to gate on. Anyone writing such code with the existing (buggy) flag would measure “no speedup on Alder Lake” and conclude the optimization wasn’t worth it, when in reality the flag would be the problem.
  • Third-party code that consumes internal/cpu indirectly (vetting tools, build-tag probes, the goexperiment machinery) gets correct information.
  • Users get a clean GODEBUG=cpu.gfni=off knob instead of having to disable AVX wholesale.

That’s a respectable list. It’s just a list of future and correctness wins, not a list of benchmarks anyone is going to publish a screenshot of.

What I take away from this

A few things, in roughly decreasing order of generality:

Feature-detection code is high-leverage and easy to get wrong. A one-line mistake in a CPUID-shaped boolean silently neutralizes every downstream optimization that ever depends on it. The blast radius is not “one buggy function”; it’s “every hot-path consumer, present and future, on every affected CPU.” There is no error message. There is no log line. The fast path simply does not fire, and the program does what it always did, only slower than it could.

Refactors that bundle “related” assignments into nested blocks are a known smell. The original code grouped GFNI with the AVX-512 setup because the EVEX-512 form of GFNI was the only thing the Go runtime cared about at the time. That was true. It was also a coincidence. When the assumption (“we only care about GFNI on AVX-512 hardware”) changed silently, because new hardware shipped and not because anyone edited Go, the nesting became wrong, and there was nothing in the code itself to flag it. Invariant tests would have caught it years ago.

Reading source code with a hardware spec open is a superpower. I would not have noticed this without having the Intel SDM tab next to the Go source tab. The line X86.HasGFNI = isSet(ecx7, cpuid_GFNI) doesn’t look suspicious in isolation. It looks suspicious only when you know that GFNI has a non-AVX-512 encoding and that Alder Lake exists. The bug lives in the gap between two correct documents.

“Why doesn’t anyone notice this in production?” is usually answered by “the bug is silent.” Performance bugs are the worst kind of bug because they look like physics. Your program is slow on Alder Lake; well, AVX-512 isn’t there, so of course it’s slow. Except the AVX-512 path wasn’t the only path that could have made it fast, and the alternative path was disabled by accident. Without a regression test that reads the truth from CPUID and compares it against what Go reports, you can never reproduce this kind of bug from a benchmark alone.

The patch is four commits totaling about 70 lines. It’s a small change. But the principle it embodies is the kind of thing I’d like to see more of throughout the runtime: feature-detection code deserves invariant tests, kill switches, and a comment explaining the hardware reality it’s mirroring.

Now that the talk prep has nerd-sniped me, I’ll probably check the other features that share GFNI’s “SSE + VEX + EVEX” shape. I already confirmed VAES is gated correctly. VPCLMULQDQ I haven’t looked at yet. There might be more.

That’s the fun of reading source code for a talk: you go in to explain a thing and you come out having changed the thing. The bugs you find weren’t placed there for you. They were just sitting there, the whole time, waiting for someone to read with intent.
