Green Tea GC: Benchmarking reality check

5 February 2026 / 7 min read
  • go
  • runtime
  • gc
  • performance
  • benchmarking

This is Part 3 of a series on Go’s Green Tea garbage collector. Part 1 covers the core insight. Part 2 covers implementation. Part 4 will cover SIMD acceleration in Go 1.26.

I’ve shown you how Green Tea works. The numbers look good: 10-40% GC CPU reduction, 50% fewer cache misses. Time to enable it and ship, right?

Not so fast. Let me tell you about DoltHub.

The cautionary tale

DoltHub runs Dolt, a SQL database with Git-like version control. Heavy on allocations, complex data structures, lots of GC pressure. A perfect candidate for Green Tea, right?

They tested it (take a look! it’s an interesting blog post). Benchmarks showed 10-40% GC improvement. Production showed… nothing. No measurable difference in real-world performance.

What happened?

Their hypothesis: less time in GC means more time hitting other bottlenecks. The memory access patterns in their application didn’t cluster well enough to benefit from span-based scanning. And subtle changes (code alignment, different GC timing) can offset gains in unexpected ways.

This isn’t a failure of Green Tea. It’s a reminder that benchmarks measure one thing, production measures everything. This is Goodhart’s Law (it again!) applied to performance: once you optimize for a metric, it stops being a good metric.

How to test

Take the following struct as the study object:

type SmallObject struct {
    ID    int64
    Name  string
    Value float64
    Tags  [4]string
}

And let’s benchmark it. Here’s where most people get tripped up. (Full source on GitHub Gist)

The gotcha: your benchmark does nothing

This looks reasonable:

func BenchmarkAllocation(b *testing.B) {
    for b.Loop() {
        obj := &SmallObject{
            ID:    42,
            Name:  "test",
            Value: 3.14,
            Tags:  [4]string{"a", "b", "c", "d"},
        }
        _ = obj
    }
}

Results:

BenchmarkAllocation-32    1000000000    0.2091 ns/op    0 B/op    0 allocs/op

Zero allocations. The compiler sees _ = obj, knows nothing escapes, and optimizes the entire allocation away. Congrats! You’re benchmarking an empty loop.

The old b.N style has the same problem:

func BenchmarkAllocationOldStyle(b *testing.B) {
    for i := 0; i < b.N; i++ {
        obj := &SmallObject{
            ID:    int64(i),
            Name:  "test",
            Value: 3.14,
            Tags:  [4]string{"a", "b", "c", "d"},
        }
        _ = obj
    }
}
BenchmarkAllocationOldStyle-32    1000000000    0.2085 ns/op    0 B/op    0 allocs/op

Same result. Zero allocations. Neither b.Loop() nor b.N magically prevents the compiler from being clever.

The second gotcha: KeepAlive isn’t enough

The usual advice is “use runtime.KeepAlive.”

So you try:

func BenchmarkSingleAllocation(b *testing.B) {
    b.ReportAllocs()
    for b.Loop() {
        obj := &SmallObject{
            ID:    42,
            Name:  "test",
            Value: 3.14,
            Tags:  [4]string{"a", "b", "c", "d"},
        }
        runtime.KeepAlive(obj)
    }
}
BenchmarkSingleAllocation-32    1000000000    3.471 ns/op    0 B/op    0 allocs/op

Still zero!

runtime.KeepAlive prevents the GC from collecting an object early, but it doesn’t force a heap allocation. The compiler’s escape analysis sees that obj never leaves the stack frame, so it stays on the stack. KeepAlive is a GC hint, not an escape hatch.

The fix: force actual escape

What you need is for pointers to actually escape. Storing objects in a heap-allocated slice does the trick:

func BenchmarkWithGCPressure(b *testing.B) {
    b.ReportAllocs()
    for b.Loop() {
        objects := make([]*SmallObject, 1000)
        for j := 0; j < 1000; j++ {
            objects[j] = &SmallObject{
                ID:    int64(j),
                Name:  "pressure-test",
                Value: float64(j) * 1.5,
            }
        }
        runtime.KeepAlive(objects)
    }
}
BenchmarkWithGCPressure-32    12468    96057 ns/op    96000 B/op    1000 allocs/op

Now we’re talking. 1000 allocations per iteration, 96KB of heap pressure. This is what actually exercises the GC.

The contrast: large objects stay on the stack

func BenchmarkLargeObjects(b *testing.B) {
    b.ReportAllocs()
    for b.Loop() {
        data := make([]byte, 8192)
        data[0] = 1
        runtime.KeepAlive(data)
    }
}
BenchmarkLargeObjects-32    288043    4111 ns/op    0 B/op    0 allocs/op

Zero heap allocations despite KeepAlive. Go stack-allocates the slice even at 8KB. This ties directly to the “large objects dominate” scenario (covered in “When to expect improvement” below). If your workload is mostly large allocations, Green Tea has nothing to optimize.

Now that you know how to write benchmarks that actually allocate, let’s test Green Tea.

Step 1: enable Green Tea

# Go 1.25 (experimental) — run benchmarks with Green Tea enabled
GOEXPERIMENT=greenteagc go test -bench=. -benchmem -count=10 -benchtime=5s > greentea.txt

# Baseline for comparison (Go 1.25 default is nogreenteagc)
go test -bench=. -benchmem -count=10 -benchtime=5s > baseline.txt

# Go 1.26+ (Green Tea is default, opt-out for comparison)
GOEXPERIMENT=nogreenteagc go test -bench=. -benchmem -count=10 -benchtime=5s > oldgc.txt

Step 2: read the GC trace

GODEBUG=gctrace=1 go test -bench=BenchmarkWithGCPressure -benchtime=5s

Output (first few cycles from BenchmarkWithGCPressure):

gc 1 @0.004s 4%: 0.17+0.61+0.045 ms clock, 5.6+0.10/1.4/0+1.4 ms cpu, 4->5->1 MB, 4 MB goal, 0 MB stacks, 0 MB globals, 32 P
gc 2 @0.005s 5%: 0.13+0.36+0.024 ms clock, 4.2+0.089/1.3/0.13+0.79 ms cpu, 3->3->1 MB, 4 MB goal, 0 MB stacks, 0 MB globals, 32 P

Notice how the heap stabilizes around 3->3->1 MB after the first cycle. The GC is keeping up with the allocation rate. The GC percentage hovers around 4-5%, showing consistent overhead.

Step 3: A/B comparison

# Compare the two runs
benchstat baseline.txt greentea.txt

-count=10 in Step 1 is crucial. In my experience, single runs tell you almost nothing. You need enough samples for statistical significance. Look for p-value <0.05 (statistically significant) and check the delta % to see if the improvement is meaningful.

Step 4: use runtime/metrics

For programmatic access:

import (
    "fmt"
    "runtime/metrics"
)

// Read GC metrics
samples := []metrics.Sample{
    {Name: "/gc/heap/allocs:bytes"},
    {Name: "/gc/heap/goal:bytes"},
    {Name: "/gc/scan/heap:bytes"},
    {Name: "/sched/pauses/stopping/gc:seconds"},
}
metrics.Read(samples)

for _, s := range samples {
    switch s.Value.Kind() {
    case metrics.KindUint64:
        fmt.Println(s.Name, "=", s.Value.Uint64())
    case metrics.KindFloat64:
        fmt.Println(s.Name, "=", s.Value.Float64())
    case metrics.KindFloat64Histogram:
        fmt.Println(s.Name, "= <histogram>")
    }
}

Sample output:

/gc/heap/allocs:bytes = 332672
/gc/heap/goal:bytes = 4194304
/gc/scan/heap:bytes = 375384
/sched/pauses/stopping/gc:seconds = <histogram>

What I find most useful is the /sched/pauses/stopping/gc:seconds histogram. It shows STW pause distribution, which Green Tea shouldn’t significantly affect. Green Tea optimizes the concurrent mark phase (the middle number in the clock times), not the STW pauses (first and last numbers).

The b.Loop() pattern (Go 1.24+)

func BenchmarkProcess(b *testing.B) {
    for b.Loop() {
        process(data)
    }
}

This replaces the old for i := 0; i < b.N; i++ pattern. But as the benchmarks above show, b.Loop() alone doesn’t prevent the compiler from optimizing away allocations. If nothing escapes to the heap, the compiler still eliminates the work. Even runtime.KeepAlive isn’t enough for a single object. You need actual escape pressure: store pointers in a heap-allocated slice, pass to an interface, or use a global sink.

When to expect improvement

Good candidates for Green Tea:

  • GC currently >5% of CPU time (check gctrace output).
  • Allocation-heavy workloads with small objects (≤512 bytes).
  • Good temporal locality (objects allocated together stay together).

Likely no improvement:

  • GC is already a tiny fraction of CPU.
  • Large objects dominate.
  • Scattered allocation patterns.
  • Other bottlenecks (I/O, network, database).

Before concluding it doesn’t help:

  • Is your benchmark representative? Micro-benchmarks often have different allocation patterns than production.
  • Did you run enough iterations? Variance >5% means unreliable results.
  • Did you check both throughput and latency? Green Tea might improve one without affecting the other.

The realistic expectation

For an application spending 10% of time in GC:

  • Best case: 40% GC improvement → 4% overall CPU reduction.
  • Typical case: 10% GC improvement → 1% overall CPU reduction.
  • DoltHub case: 0% real-world improvement.

These numbers might seem small, but at scale they matter. A 1-4% CPU reduction across a fleet of servers is real money. Just don’t expect Green Tea to solve your performance problems if GC isn’t actually the bottleneck.

My recommendation

Test on your actual workload before enabling Green Tea. Profile first. If GC isn’t showing up, Green Tea won’t help. Build both versions, run the same workload, compare with benchstat.

Test in staging before production. Different GC timing can surface latent bugs and race conditions. Keep monitoring after rollout.

Green Tea is a good improvement to Go’s GC, not magic. Applications that benefit most are already GC-bound with favorable allocation patterns. For everyone else, it’s a modest improvement you might not notice. And that’s fine.
