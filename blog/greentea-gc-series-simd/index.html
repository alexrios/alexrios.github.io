Part of series: Green Tea GC

Green Tea GC: SIMD acceleration

16 May 2026 / 9 min read
  • go
  • runtime
  • gc
  • performance
  • simd

This is Part 4 of a series on Go’s Green Tea garbage collector. Part 1 covers the core insight. Part 2 covers implementation. Part 3 covers benchmarking.

Intel built VGF2P8AFFINEQB to make AES encryption faster. The Go team noticed it could expand bitmaps in one instruction and quietly drafted it into the garbage collector. Hardware sometimes surprises you.

That single repurposed instruction is the heart of Go 1.26’s SIMD-accelerated GC: another 10-20% reduction in GC overhead on top of Green Tea’s existing gains.

What strikes me about this story isn’t the instruction itself. It’s that the optimization was sitting there the whole time, waiting for someone to notice the data was finally regular enough to vectorize.

The setup nobody planned

Three of Go’s GC decisions, made long before SIMD was on the table, accidentally aligned. Non-moving collection keeps objects in their spans for life, so span contents stay stable across the whole mark phase. Segregated-fits allocation packs every object in a span into one size class, at fixed offsets, with a uniform pointer layout. And span-based scanning (Green Tea) organizes work into 8 KB chunks instead of arbitrary pointer-chases.

None of these were chosen with vectorization in mind. Non-moving simplified the compiler interface. Segregated-fits made allocation O(1). Green Tea was about cache locality.

But put them together and you get regular, predictable data: packed mark bits at known offsets, uniform pointer layouts per span, no race against object relocation. That’s exactly the shape SIMD needs.

I’ve noticed that constraints often enable optimizations rather than just limiting them. Go’s “we don’t move objects” rule is what makes this whole thing possible. Try doing this on G1 or ZGC, where objects can be evacuated mid-cycle. Span contents wouldn’t be stable enough to vectorize.

Why the old GC couldn’t be vectorized

The traditional GC scanned objects one at a time. Each object had a different size, lived at a different address, and had a different pointer layout. There was no regularity to feed to a vector unit, and SIMD without regularity is just expensive scalar code.

Green Tea changed the work unit. Within a span, all objects are the same size, at sequential addresses, with the same pointer bitmap. That’s the precondition for everything below.

The algorithm in five steps

When a span with multiple marked objects is scanned, the AVX-512 path runs roughly this:

Step 1: load mark and scan bits

Pull the span’s mark bits and scan bits into 512-bit AVX-512 registers. Each register holds 512 bits, one per object. A span can hold up to 1024 objects (8 KB / 8-byte minimum), so two register loads cover the worst case.

Step 2: compute what’s left to scan

toScan = marks &^ scans

Marked but not yet scanned. One instruction. 512 objects at a time.

Step 3: expand the bitmap

This is the interesting part.

Mark bits track objects: 1 bit per object. But to find pointers, we need to know which 8-byte words to look at. A 48-byte object is 6 words, so its single mark bit has to expand into 6 word-bits:

Object size: 48 bytes (6 words)
Mark bit:    1
Expanded:    111111

Different size classes expand at different ratios. The Go build generates 26 specialized expander functions, one per eligible Green Tea size class, each with a custom Galois Field matrix optimized for its expansion factor (source: mkasm.go). More on the matrix below.

Step 4: load the pointer bitmap

Go’s type system already knows which words in an object hold pointers. Load that bitmap for the span’s size class.

Step 5: intersect and collect

activePointers = expandedMarks & pointerBitmap

Now we know exactly which words are live pointers. Walk them out 64 bytes at a time using VPCOMPRESSQ, which packs the masked qwords contiguously into the output buffer. No branchy pointer-by-pointer collection.

That’s the whole loop. The full assembly lives in scan_amd64.s, but the shape is just load → mask → expand → intersect → compress → store.

A surprising real-world quirk, from comments in that same file:

// N.B. VPCOMPRESSQ supports a memory destination. Unfortunately, on
// AMD Genoa / Zen 4, using VPCOMPRESSQ with a memory destination
// imposes a severe performance penalty of around an order of magnitude
// compared to a register destination.
//
// ... (workaround caveat on other microarchitectures elided)
//
// AMD Turin / Zen 5 fixes this issue.

The runtime issues VPCOMPRESSQ to a register, then stores separately. Slower in theory, dramatically faster on current AMD silicon. The ISA tells you what’s legal; benchmarks tell you what’s fast. (Daniel Lemire’s write-up of the Zen 4 gotcha has the measurements if you want them.)

The magic instruction: VGF2P8AFFINEQB

The expansion step is the algorithmic bottleneck. Naive bit expansion is loops and conditionals with a variable per-class ratio, exactly the kind of thing SIMD hates.

VGF2P8AFFINEQB does an 8×8 bit affine transformation in GF(2): every output bit is the XOR of selected input bits, plus an optional constant. With the constant set to zero (as Go uses it), it’s pure matrix-vector multiplication over GF(2), and the matrix is just a table of which input bits feed each output bit.

Feed it the right matrix and “expand 1 mark bit into N word bits” becomes one instruction. For an entire 64-byte chunk of the span.

The matrices are generated at Go build time by mkasm.go:

// From /src/internal/runtime/gc/scan/mkasm.go
xf := int(ob) / 8                       // expansion factor (words per object)
for oBit := 0; oBit < outputBits; oBit++ {
    iBit := oBit / xf                    // which input bit feeds this output
    byteMat.mat[oBit%8] = 1 << (iBit % 8)
}

Each eligible size class gets its own pre-computed matrix bound to its own expander. At runtime, the right expander is called via a function-pointer table indexed by size class.

Why does this instruction exist? Cryptography. VGF2P8AFFINEQB was added in AVX-512 for accelerating AES, SM4, and other ciphers built on Galois Field arithmetic. The Go team reframed bit expansion as matrix multiplication in GF(2) and quietly inherited a piece of hardware designed for something completely different.

This is mechanical sympathy taken to its conclusion: a data-structure problem (sparse bitmap expansion) becomes a linear algebra problem, and AVX-512 already has the kernel for it.

When SIMD wins, and when it doesn’t

In my experience, the vectorized path isn’t free. Loading masks, configuring registers, running the compress loop: there’s setup overhead. For sparse spans, scanning the handful of marked objects directly is faster.

The runtime checks two things before taking the SIMD path (mgcmark_greenteagc.go):

if !scan.HasFastScanSpanPacked() || objsMarked < int(nelems/8) {
    scanObjectsSmall(spanBase, elemsize, nelems, gcw, &toScan)  // scalar
    return
}
// otherwise: ScanSpanPacked (SIMD)
ConditionPath
No AVX-512 extensionsScalar
AVX-512 available, < 12.5% of objects markedScalar
AVX-512 available, ≥ 12.5% of objects markedSIMD

The 12.5% threshold (nelems/8) is where SIMD setup pays for itself. Sparser than that and the scalar path wins.

The runtime tracks these paths separately for observability, which is what lets you tell whether your workload is actually feeding the vector unit:

type sizeClassScanStats struct {
    spansDenseScanned     uint64 // scanned with ScanSpanPacked
    spanObjsDenseScanned  uint64
    spansSparseScanned    uint64 // scanned with scanObjectsSmall
    spanObjsSparseScanned uint64
    sparseObjsScanned     uint64
}

Hardware requirements

The AVX-512 kernel needs five specific extensions, checked at startup via CPUID, with automatic fallback to scalar if any are missing:

var avx512ScanPackedReqsMet = cpu.X86.HasAVX512VL &&
    cpu.X86.HasAVX512BW &&
    cpu.X86.HasGFNI &&
    cpu.X86.HasAVX512BITALG &&
    cpu.X86.HasAVX512VBMI
ExtensionRole
AVX512VLVector length flexibility (128/256/512-bit)
AVX512BWByte and word operations on bitmaps
GFNIVGF2P8AFFINEQB, the magic instruction
AVX512BITALGVPOPCNTB for per-byte population count
AVX512VBMIVPERMB for byte shuffling

CPUs with all five:

  • Intel Ice Lake (client 2019+, 3rd Gen Xeon Scalable 2021+)
  • Intel Tiger Lake and newer client CPUs (2020+)
  • AMD Zen 4 (EPYC 9004 “Genoa”, Ryzen 7000, 2022+)

If you’re on older silicon, Apple Silicon, ARM servers, or anything without GFNI, you get the scalar path. Green Tea’s base gains still apply; the SIMD bonus doesn’t.

Could other GCs do this?

I think it’s worth asking. Java’s modern collectors all face the same memory-wall problem Green Tea solves. So why hasn’t G1 or ZGC shipped something similar?

Because they move objects.

G1 evacuates live data between regions. ZGC relocates concurrently. Shenandoah does concurrent compaction. In all three, a region’s contents are unstable across (and during) GC cycles. You can’t enqueue a region, wait, and assume the same objects will still be there with the same layout when you dequeue it.

Go’s non-moving design, historically dismissed as a fragmentation risk, is the structural property that makes span-level SIMD work at all. The mark bits stay meaningful. The objects stay put. The pointer bitmap stays valid. The vector unit gets the regularity it needs.

This isn’t a brag about Go’s GC. It’s the cascade I started with. An early-Go simplicity decision turned out to enable a 2025-era optimization that nobody planned for.

Performance

Measured improvements from Go 1.26 development:

  • Prototype benchmarks: 15-20% additional GC reduction on top of base Green Tea
  • Production target: 10% additional (conservative, accounting for real-world variance)

Combined with base Green Tea:

  • Without SIMD: 10-40% GC improvement vs traditional GC
  • With SIMD on supported hardware: 20-50% total improvement on favorable workloads

The reasons for the wide range should be familiar from Part 3:

  • Only CPUs with all five extensions get the SIMD path
  • Sparse spans fall back to scalar regardless of CPU
  • Benefits concentrate in the 8-512 byte size classes
  • Allocation-heavy workloads see larger gains

And as DoltHub demonstrated, GC improvements don’t translate one-for-one into application improvements. If GC is 10% of CPU and SIMD cuts it in half, the wall-clock win is 5%. Often it’s lost in the noise of I/O, network, and database latency. Measure before you celebrate.

My take: SIMD acceleration in Go 1.26 is the third step in a cascade nobody designed.

  1. Non-moving collector (early Go, 2009): objects stay in spans forever
  2. Span-based scanning (Green Tea, 2024): exploit that stability for cache locality
  3. SIMD vectorization (Go 1.26): exploit the resulting regularity with AVX-512

Each step looked like it was about something else. None of them planned for the next. Green Tea wasn’t framed as “make the data regular enough to vectorize”, but that’s what it did.

I find that the pattern shows up everywhere in systems work: constraints are options for later. Fixed layouts enable vectorization. Stable data enables batching. Predictable access enables prefetching. Every limitation your system has is a structural property the next optimization can stand on.

Sometimes the best optimization is making your data easier to optimize.
