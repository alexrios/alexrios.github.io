Part of series: Go 1.25

The belated discovery of basic ergonomics

18 June 2025 / 3 min read
  • go
  • ergonomics
  • sync

Go has made a startling discovery: perhaps, just perhaps, developers shouldn’t have to write the same three lines of code every single time they want to spawn a goroutine and wait for it. Revolutionary stuff, really.

The old dance of tedium

For over a decade, Go developers have been performing what I like to call “The WaitGroup Waltz”. A delightful little dance that goes something like this:

var wg sync.WaitGroup
wg.Add(1)
go func() {
    defer wg.Done()
    doSomething()
}()
wg.Wait()

Charming, isn’t it? Three lines to express what should be a single concept. It’s like having to separately declare your intention to walk, lift your foot, move it forward, and put it down again, every single time you want to take a step.

The enlightenment of Go 1.25

But hark! The clouds have parted, and Go has blessed us with WaitGroup.Go():

var wg sync.WaitGroup
wg.Go(doSomething)
wg.Wait()

Gasps audibly. They’ve managed to reduce three lines to one. At this rate of innovation, we might see structured concurrency by 2030.

What’s actually happening here

The new Go() method is essentially a convenience wrapper that does what every Go developer has been doing manually since time immemorial:

  1. Calls Add(1) to increment the counter
  2. Spawns a goroutine
  3. Ensures Done() is called when the function returns (via defer)

It’s the programming equivalent of finally getting a dishwasher after years of washing dishes by hand. Technically, the old way worked, but why would you want to keep doing it?

The subtle brilliance (yes, really)

Now, before you think I’m being entirely dismissive, there’s actually some thoughtful design here. The method signature is delightfully simple: func (wg *WaitGroup) Go(f func()). No fancy generics, no complex parameters. Just a function that takes a function. It’s Go’s version of sophistication: profound simplicity that makes you wonder why it took so long.

The panic requirement (f must not panic) is particularly delicious. It’s Go’s way of saying “we’ll make this convenient for you, but if you mess up, we’re not cleaning up your mess.”

Memory model implications

The documentation carefully notes that the function return “synchronizes before” any Wait() call it unblocks. This is Go’s way of saying “we’ve thought about the scary concurrent memory stuff so you don’t have to.” It’s the same guarantee you had before, just now wrapped in a more palatable package.

The productivity paradox

Here’s what’s genuinely interesting: this tiny change will probably save millions of lines of boilerplate code across the Go ecosystem. It’s a perfect example of how small ergonomic improvements can have outsized impacts. Sometimes the best innovations aren’t the flashy new features. They’re the ones that eliminate the thousand tiny annoyances that accumulate like dust in the corners of your codebase.

The bigger picture

This change represents something important in Go’s evolution: an acknowledgment that developer experience matters. For years, Go’s philosophy seemed to be “if it’s possible to do it in the most explicit way, that’s the only way we’ll support.” But WaitGroup.Go() suggests a subtle shift toward “maybe we should make the common case easy.”

It’s a small step, but in the world of Go’s glacial pace of change, it’s practically a sprint.

Final thoughts

Will WaitGroup.Go() change the world? Hardly. Will it make Go code slightly more pleasant to write and read? Absolutely. And sometimes, that’s exactly the kind of progress we need.

Now, if they could just apply this same thinking to structured concurrency…

But that’s a rant for another day.
